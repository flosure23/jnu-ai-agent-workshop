# 개발 계획 (체크리스트)

`spec.md`(제품 명세)와 `api-spec.md`(데이터 주고받기)를 기준으로, **실제 작업 순서**를 단계별로 정리했다.  
완료한 항목은 `- [ ]` 를 `- [x]` 로 바꿔 쓰면 된다.

---

## 전제

- [ ] `spec.md` · `api-spec.md`를 한 번 훑어본다.
- [ ] Supabase 프로젝트가 있고, 대시보드에 접속할 수 있다.

---

## 1단계 — 프로젝트·Supabase 뼈대

**목표:** Next.js에서 Tailwind·환경 변수·Supabase 클라이언트를 쓸 수 있게 한다.

### 1-1. Next.js 프로젝트

- [ ] Next.js 생성 (**App Router**, **TypeScript**, **Tailwind** 포함).
- [ ] 루트 `layout` / 기본 페이지가 빌드·실행되는지 확인.

### 1-2. Supabase 연동 (코드)

- [ ] `.env.local`에 `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` 설정.
- [ ] `.gitignore`에 `.env.local` 포함 확인 (`create-next-app` 기본값이지만 한 번 체크).
- [ ] `.env.example` 생성 — 키 이름만 적고 값은 비워 둔다 (나중에 다시 세팅할 때 참고용).
- [ ] 브라우저용 / 서버용 **Supabase 클라이언트** 생성 (예: `lib/supabase/client.ts`, `server.ts` 등 프로젝트 규칙에 맞게).

### 1-3. DB 테이블 `entries`

- [ ] `spec.md`의 **DB 스키마**대로 `entries` 테이블 생성 (마이그레이션 또는 SQL 실행).
- [ ] `user_id` → `auth.users` FK, 컬럼·타입이 명세와 일치하는지 확인.

### 1-4. (선택) 로컬에서 DB만 먼저 테스트

- [ ] 대시보드 **Table Editor** 또는 간단한 쿼리로 `entries`에 테스트 행을 넣어 볼 수 있는지 확인.

### 1-5. TypeScript 타입 자동 생성

- [ ] `supabase gen types typescript` 등으로 DB 스키마에서 **타입 파일**을 생성해 둔다 (예: `lib/database.types.ts`).
- [ ] 이후 단계에서 `entries` 행 타입을 직접 만들 필요 없이 이 파일을 import 해서 쓴다.

## 2단계 — 인증 (US-1)

**목표:** 회원가입·로그인·로그아웃·비밀번호 재설정(간단) + 비로그인 시 보호 구역 차단.

### 2-1. 인증 레이아웃 + 화면

- [ ] `(auth)/layout.tsx` — GNB 없는 **인증 전용 레이아웃** (센터 정렬 등). 프로젝트에 전역 CSS가 있다면 그 스타일을 참고한다.
- [ ] `signup` — 이메일·비밀번호, 유효성·에러 메시지.
- [ ] `login` — 로그인 + **비밀번호 찾기** 링크.
- [ ] `forgot-password` — 이메일만 받고 `resetPasswordForEmail` 호출.
- [ ] `reset-password` — 메일 링크 후 **새 비밀번호** 입력 (`updateUser`).
- [ ] 필요 시 `auth/callback` — Supabase 리다이렉트·세션 처리 (문서대로 **필요할 때만**).

### 2-2. Supabase 대시보드 설정

- [ ] **Site URL** · **Redirect URLs**에 로컬/배포 URL과 `/reset-password` 등 허용 경로 등록.
- [ ] Authentication > Settings에서 **"Confirm email" OFF** (MVP: 가입 즉시 로그인 가능하게).

### 2-3. 인증 가드 + GNB 뼈대

- [ ] `(main)` 레이아웃에서 **비로그인이면 로그인으로 리다이렉트**.
- [ ] GNB 자리: 로고, 새 일기, 로그아웃 (스타일은 나중에 다듬어도 됨).

---

## 3단계 — 일기 작성 (US-2)

**목표:** 로그인한 사용자만 본인 `user_id`로 `entries`에 INSERT.

- [ ] `entries/new` — 제목·본문·날짜 폼 (날짜 기본값 오늘).
- [ ] 유효성: 제목·본문 필수.
- [ ] 저장 시 `user_id` = **현재 세션 사용자 UUID** (`api-spec.md` 공통 규칙).
- [ ] 성공 시 토스트(또는 임시 알림) + 목록으로 이동.
- [ ] 실패 시 에러 메시지 표시.

---

## 4단계 — 일기 목록 + 상세 (US-5 + US-6)

**목표:** 본인 일기만, `date DESC`, 페이지네이션, 상세·404.

### 4-1. 목록

- [ ] `entries` 목록 — 카드 UI, **본인만** `.eq('user_id', …)`.
- [ ] 정렬: **`date` 내림차순**.
- [ ] **페이지네이션** — **한 페이지 10건**, `range` 등 (`api-spec.md` 참고).
- [ ] 빈 목록일 때 안내 문구 + 작성 버튼.
- [ ] 로딩(스켈레톤/스피너).

### 4-2. 상세

- [ ] `entries/[id]` — 제목·날짜·본문 전체.
- [ ] 본인 글이 아니거나 없으면 **404** 처리.
- [ ] 수정·삭제 버튼 자리, 목록으로 가기 링크.
- [ ] 로딩 표시.

---

## 5단계 — 일기 수정 + 삭제 (US-3 + US-4)

**목표:** UPDATE / DELETE 시에도 항상 본인 `user_id` 조건.

- [ ] `entries/[id]/edit` — 기존 값 채운 폼, UPDATE.
- [ ] 성공/실패 피드백 (토스트).
- [ ] (명세) `updated_at` 없음 — 최신 내용은 **새로고침** 또는 다시 들어와서 확인.
- [ ] 상세에서 삭제 — 확인 모달 후 DELETE, 성공 시 목록으로 이동.

---

## 6단계 — 검색 (US-7)

**목표:** 목록 상단 검색, 제목·본문 `ilike`, 디바운스 300ms.

- [ ] 검색어 입력 후 **300ms 디바운스** 뒤 재조회.
- [ ] `user_id` 필터 유지 + 제목/본문 OR 조건 (`api-spec.md`).
- [ ] 정렬·페이지네이션은 목록과 동일 규칙.
- [ ] 결과 없음: “검색 결과가 없습니다”.

---

## 7단계 — 공통 UI·마무리

**목표:** `spec.md` 공통 기능·UX 정리.

- [ ] **토스트** — 저장·수정·삭제 성공/실패 공통 컴포넌트.
- [ ] **확인 모달** — 삭제 등에서 쓰는 공통 컴포넌트 ("정말 삭제하시겠습니까?" 등).
- [ ] **GNB** 스타일·반응형 정리 (모바일/데스크톱).
- [ ] `loading.tsx` / `error.tsx` 필요한 라우트에 배치.
- [ ] 인증·폼·목록 전반 **에러 메시지** 사용자에게 보이기.
- [ ] 한 번 끝에서 끝으로 시나리오 테스트 (가입 → 작성 → 목록 → 상세 → 수정 → 삭제 → 검색 → 로그아웃).
