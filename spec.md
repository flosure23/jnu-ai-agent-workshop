## 2. 유저 스토리 & 기능 명세

### US-1. 회원가입 / 로그인

> *사용자로서, 나만의 계정을 만들어 로그인하고 싶다. 다른 사람이 내 일기를 볼 수 없어야 한다.*

| 구분 | 내용 |
|------|------|
| 인증 | 이메일 + 비밀번호 회원가입 (**Supabase Auth**) |
| 이메일 인증 | MVP에서는 **OFF** — 가입 즉시 로그인 가능 |
| 세션 | 로그인 / 로그아웃 |
| 비밀번호 재설정 | 포함. **간단 구현**은 아래 문단 참고 |
| 접근 제어 | 비로그인 시 로그인 페이지로 리다이렉트 (**인증 가드**) |
| 입력 검증 | 이메일 형식, 비밀번호 최소 길이 |
| 피드백 | 에러 메시지 (중복 이메일, 잘못된 비밀번호 등) |
| **데이터 소유** | DB·API에서 일기는 **항상** `user_id = 현재 로그인 사용자`로만 조회·저장한다. (RLS 없음, **앱·쿼리에서 강제**) |

#### 비밀번호 재설정이란

로그인 비밀번호를 잊었을 때 쓰는 기능이다. 사용자가 **“비밀번호 찾기”**에 이메일을 넣으면 Supabase가 **그 주소로 메일**을 보내고, 메일 안 **링크를 누르면** 우리 앱의 비밀번호 재설정 화면으로 들어와 **새 비밀번호**를 입력한다.

#### 비밀번호 재설정 — 간단 구현 방향

- **Supabase Auth 기본 흐름만** 쓴다: `resetPasswordForEmail`(이메일 요청) → 메일 링크 클릭 → 세션/해시 처리 후 `updateUser`로 새 비밀번호 저장. 별도 백엔드 API는 만들지 않는다.
- **화면은 최소 2단계**: (1) 이메일만 받는 “찾기” 화면 — 로그인 페이지에서 링크로 연결 가능, (2) 메일 링크로 들어온 뒤 **새 비밀번호 입력**만 하는 `reset-password` (또는 동일 페이지에서 단계만 구분).
- **Supabase 대시보드**: Site URL·Redirect URLs에 우리 앱 도메인과 비밀번호 재설정 후 돌아올 경로(예: `/reset-password`)를 등록한다. 로컬 개발용 `http://localhost:3000/...`도 함께 넣는다.
- **이메일 발송**: MVP는 Supabase **기본 메일**(또는 대시보드에서 켜 둔 SMTP)으로 충분하고, 브랜딩·커스텀 템플릿은 나중에 한다.
- 링크로 들어온 직후 **해시/세션 교환**이 필요하면 Next.js에서 `auth/callback` 같은 **짧은 라우트 한 개**만 두고, 나머지는 Supabase 클라이언트 문서 패턴을 따른다.

### US-2. 일기 작성

> *사용자로서, 오늘 하루를 기록하고 싶다.*

- 제목, 본문, **날짜** 입력 폼
- 날짜 **기본값**: 오늘 날짜 자동 설정
- 유효성 검사: 제목·본문 **필수**
- **Supabase DB**에 저장
- 저장 성공: **토스트** 알림 + 목록으로 이동
- 저장 실패: 에러 메시지 표시

---

### US-3. 일기 수정

> *사용자로서, 이미 작성한 일기를 고치고 싶다.*

- 기존 내용이 채워진 **수정 폼**
- DB **UPDATE**로 내용 반영
- 성공/실패 **토스트**
- **수정 시각(`updated_at`) 자동 기록은 MVP에서 생략** — 화면에 최신 내용을 보이려면 **새로고침**하거나 목록·상세로 다시 들어오면 된다.

---

### US-4. 일기 삭제

> *사용자로서, 필요 없는 일기를 삭제하고 싶다.*

- 삭제 버튼 + **확인 모달** (“정말 삭제하시겠습니까?”)
- **하드 삭제**
- 성공/실패 **토스트**
- 삭제 후 **목록 페이지**로 리다이렉트

---

### US-5. 일기 목록 조회

> *사용자로서, 내가 쓴 일기들을 최신순으로 보고 싶다.*

- **카드** 형태 목록 (제목, 날짜, 본문 미리보기)
- 정렬: 일기 **작성일** 컬럼 `date` 기준 **최신순** (`date DESC`)
- 목록은 **페이지네이션**만 사용 (무한 스크롤 없음)
- 빈 상태: “첫 일기를 작성해보세요” + 작성 버튼
- 로딩: 스켈레톤 또는 스피너

---

### US-6. 일기 상세 조회

> *사용자로서, 일기 하나를 선택해 전체 내용을 읽고 싶다.*

- 상세 페이지: 제목, 날짜, 본문 전체
- **수정 / 삭제** 진입점
- 목록으로 돌아가기 네비게이션
- 없는 일기 접근 시 **404**
- 로딩 상태 표시

---

### US-7. 일기 검색

> *사용자로서, 과거 일기를 키워드로 찾고 싶다.*

- 검색 바: **목록 페이지 상단**
- **제목 + 본문** 키워드 검색 (**Supabase `ilike`**)
- **디바운스** 300ms
- 결과 없음: “검색 결과가 없습니다”

## 4. 페이지 구조 (App Router)

```
app/
├── (auth)/
│   ├── login/page.tsx              # 로그인 (+ 비밀번호 찾기 링크)
│   ├── signup/page.tsx             # 회원가입
│   ├── forgot-password/page.tsx    # 비밀번호 찾기 — 이메일만 (간단 구현)
│   ├── reset-password/page.tsx     # 메일 링크 후 새 비밀번호 입력
│   └── auth/callback/route.ts      # 필요 시만: Supabase 리다이렉트·세션 교환
├── (main)/
│   ├── layout.tsx                  # 인증 가드 + GNB
│   ├── entries/
│   │   ├── page.tsx                # 목록 + 검색 (US-5, US-7)
│   │   ├── new/page.tsx            # 작성 (US-2)
│   │   └── [id]/
│   │       ├── page.tsx            # 상세 (US-6)
│   │       └── edit/page.tsx       # 수정 (US-3)
└── layout.tsx                      # 루트 레이아웃
```

## 5. DB 스키마 (MVP)

### 테이블: `entries`

| 컬럼 | 타입 | 제약 |
|------|------|------|
| `id` | `uuid` | PK, 기본값 `gen_random_uuid()` |
| `user_id` | `uuid` | FK → `auth.users`, `NOT NULL` |
| `title` | `text` | `NOT NULL` |
| `content` | `text` | `NOT NULL` |
| `date` | `date` | `NOT NULL`, 기본값 `CURRENT_DATE` — 목록 정렬·“작성일” 표시에 사용 |
| `created_at` | `timestamptz` | 기본값 `now()` |
